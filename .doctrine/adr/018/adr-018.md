# ADR-018: Daemon topology and control-plane transport for the client end-state

> **Follows ADR-017.** ADR-017 decided the destination (Emacs is a client;
> the trust boundary is a protocol property) and deliberately scheduled
> nothing. This ADR answers two of its three open questions and fixes the
> topology and transport that the migration will actually land in. It does not
> re-argue the end-state; if ADR-017 is rejected, this ADR falls with it.

## Context

Since ADR-017 and RFC-001 were written (2026-07-06), RFC-001 D3 has landed:
SL-012 extracted SATAN from `.emacs.d` into this repository. The ownership cut
is done; the language/process cut is not. Working in the extracted tree
supplies evidence that was not available when ADR-017 was drafted:

- **The attrd bridge is the cost of deferring the topology decision.**
  `satan-attribute.el` enqueues outcome payloads into `satan_outcome_inbox` +
  `pg_notify`; `satan-attribute-listener.el` (372 lines) runs a dedicated
  `satan-attrd notify-stream` subprocess — a binary that exists *only* because
  a `psql` pipe will not surface async notifications — consumes
  `satan_audit_inbox`, validates, appends to `transcript.jsonl`, and replies
  through a third table `satan_audit_replies`. Two queues, a claim/reply
  protocol, a bespoke transport binary and a wire-shape version stamp, so that
  a Rust process may mutate state on behalf of an Emacs process that no longer
  holds the state. This is ADR-017's "transition risk: split authority",
  already realised.
- **Policy is unreadable outside an Emacs image.** `satan/satan-mode.el:80-208`
  is the whole mode/tool/capability/budget table, held as `defcustom` data. No
  daemon, CLI or harness can answer "what may mode `motd` call?" without
  booting Emacs. ADR-017 left the owner of this table explicitly open.
- **The registry already has a second consumer.** `satan/satan-tools.el` does
  argument validation, capability gating and JSON-Schema emission;
  `satan/satan-mcp.el` serves that schema over a UDS MCP socket to interactive
  pi sessions. Emacs is already acting as a policy *server* — pointed the wrong
  way.
- **Almost no tool handler uses the editor as an editor.** Under POL-001's
  test: only `satan-tools-org.el` (14 org call sites) and one call in
  `satan-tools-atsatan.el` touch org/denote APIs. Every other handler is file
  reads, `psql` subprocesses, git, IPC, D-Bus or a shell-out. POL-001's
  "earns the seat" list was written when the broker's presence made colocation
  free; the extraction has made the incidental tenancy visible.
- **POL-001 trigger 1 has fired, and the copy-instead-of-extract failure has
  already happened once.** When a second entry point (MCP / interactive) needed
  run identity, the run-context layer was cloned rather than extracted:
  `satan-run.el` and `satan-broker.el` each define `cl-defstruct satan-run`
  with identical slots, each define `defcustom satan-runs-dir` and
  `satan-hippocampus-dir`, and each carry a forked `mint-id` / `prepare` /
  `dir-for-id` / `tool-ctx`. In `satan-run.el:27` the struct generates accessor
  `satan-run-prepare`; `satan-run.el:77` then defines
  `(defun satan-run-prepare (mode) ...)` over the top of it. Verified
  empirically: the collision is currently **latent** — cl-defstruct accessors
  inline at load time, so existing callers compiled to `(aref run-ctx 18)`
  before the clobber — but a direct `(satan-run-prepare run-struct)` silently
  returns a freshly minted `run_id` with an unfrozen `time_now` instead of the
  frozen prepare plist, with no error. Its disarming depends on load order and
  on slot index 18 matching across two independent struct definitions.
- **The model client already left.** `satan/harness/` is a Python
  OpenAI-compatible runloop speaking the JSONL protocol on stdio. `gptel`
  survives as a binary name (`jailed-satan-gptel-harness`) and nothing else.

## Decision

**D1 — One daemon repository.** The headless core of ADR-017 §1 lives in a
single repository, expected to become a Cargo workspace of several crates.
Whether it ships as one executable or several, and whether it runs as one
process or several, is deliberately **not** decided here — that is driven out
by the execution model (OQ-1). `satan-attrd` is absorbed by that repository
rather than standing beside it as a peer authority: attributes, patterns,
outcomes, traces and hypotheses share one database, and splitting authority
across two daemons rebuilds today's problem with better syntax. This is the
concrete shape of RFC-001 D2's composition root: core repo + client(s) +
protocol, composed by the `satan` flake.

**D2 — The daemon core owns policy and the tool registry; every surface reads
it.** *(Resolves ADR-017 open question 2.)* Mode/tool allowlists, capabilities,
risk levels, budgets and jail profiles are owned by the core and published as a
read-only artifact that the Emacs client, the harness and any future surface
read rather than restate. Rationale: this is the authority item most likely to
be silently duplicated (it is data, so copying is easy and looks harmless), and
ADR-017 §3's single-owner ledger is unverifiable while the table can only be
read by booting an editor. A dedicated policy store separate from the core was
considered and rejected as a third component owning nothing else.

**D3 — Postgres is a state store, not a bus; the control plane is explicit
RPC.** The `satan_outcome_inbox` / `satan_audit_inbox` / `satan_audit_replies`
+ `pg_notify` + `notify-stream` arrangement is transitional and will be
retired. This refines rather than contradicts RFC-001 D5 ("durable logs over
RPC"): the **data plane** — perception, segments, evidence, audit — keeps the
panopticon shape of durable append-only logs read by consumers. The **control
plane** — capability decisions, run lifecycle, ledger handoff, kill switches —
is inherently request/response, and today it is a chatty RPC *smuggled through
a database*, paying queue costs for RPC semantics. It becomes a direct RPC over
a Unix domain socket. Durable state stays in Postgres.

**D4 — Sequencing is policy-first, execution-last.** Migration remains gated by
POL-001 triggers per ADR-017 §5; where triggers permit a choice, the order is:

1. **Rehearse in-language.** Collapse the `satan-run.el` / `satan-broker.el`
   duplication to a single run-context owner and rename the colliding defun
   (`satan-run-prepare` → `satan-run-new-ctx`). Same language, low risk, and it
   proves the single-owner discipline before any authority item migrates.
2. **Policy + registry (D2).** Pure data plus pure functions, no I/O; the first
   entry in the ADR-017 §3 authority ledger.
3. **Handlers by substrate, opportunistically.** Under POL-001's test the
   editor-as-editor set is smaller than its "earns the seat" list assumes, but
   nothing outside that set has an Emacs dependency whose removal pays off
   urgently.
4. **Run execution last.** ~980 lines of spawn / filter / sentinel / finalize /
   budget / failure-announce in `satan-broker.el`. Largest, best-tested, least
   architecturally contested; moving it is a quality win, not a boundary fix.

**D5 — No new stateful layer lands in elisp.** The hypothesis board, probe
records and pattern records are daemon-owned from the start. Pattern state is
already Postgres-backed; building these in elisp now would guarantee a second
extraction and would enlarge exactly the half POL-001 warns about growing.

**D6 — POL-001 is re-homed here, unchanged.** Its earns-the-seat test survives
as the per-module gate; this ADR supplies the destination and the order, not a
new test. Its "earns the seat" and anti-candidate lists were written against
`.emacs.d` paths and pre-extraction assumptions and need a mechanical pass, not
a re-argument.

## Consequences

### Positive

- The authority ledger ADR-017 §3 requires becomes checkable, because policy
  has a definition readable outside the editor.
- The attrd bridge becomes deletable rather than the template the next daemon
  copies.
- The seed brief's hypothesis / probe / pattern work has an owner on day one.
- RFC-001 D2's composition root gains a concrete shape to compose.
- Run execution eventually stops being hostage to the editor's lifecycle
  (`satan-tools-agenda.el:1-11` currently uses `timeout(1)` as a supervisor so
  a stalled `gcalcli` cannot freeze the editor).

### Negative

- **D4.2 must land as a cutover, not an addition.** The `satan-run.el` clone is
  proof this project fails that way by default; a duplicated policy table would
  be materially worse than a duplicated struct.
- An RPC surface is a new security boundary to design (OQ-3); today privilege
  separation is partly implicit in "it is all one trusted Emacs".
- Cross-process debugging replaces in-image debugging past the policy boundary.
- Elisp test coverage for migrated layers has to be re-earned in Rust; the
  broker suite is the strongest asset being disturbed.
- POL-001's list pass (D6) is unglamorous work that blocks nothing and will
  therefore tend to be skipped.

### Neutral

- Language is not the criterion anywhere in this ADR; handlers stay elisp until
  POL-001's test moves them.
- The Python harness is unaffected — the JSONL protocol is the membrane either
  way, and `satan/protocol/fixtures.json` keeps both validators honest.

## Verification

- **VT-1** — A non-Emacs process answers "what tools may mode `motd` call, at
  what risk, requiring what capability?" from the same source the broker uses.
  Acceptance test for D2; currently false.
- **VT-2** — Exactly one definition of the `satan-run` struct, of
  `satan-runs-dir` and of `satan-hippocampus-dir` exists in the tree, and no
  symbol is defined both as a struct accessor and as a `defun`. Enforceable as
  a lint check alongside `bin/elisp-locate-paren-error`. Acceptance test for
  D4.1.
- **VT-3** — After D3, no `pg_notify` channel and no `*_inbox` table carries
  control-plane traffic, and the `notify-stream` subprocess is gone. Grep-able.
- **VA-4** — Any new stateful layer (hypothesis, probe, pattern) has no elisp
  state module. Checked at design review per D5.
- **VH-5** — ADR-017's absent-client test and no-dual-enforcement check are run
  at each migration audit; this ADR adds no separate ceremony.

## Open questions

- **OQ-1** — Execution model: does the core supervise runs (spawning the jailed
  harness itself), or does it remain a policy/state authority that surfaces
  call while they spawn? Determines process and executable count under D1;
  settle before D4.4.
- **OQ-2** — RPC surface shape under D3: MCP over UDS (reusing `satan-mcp.el`'s
  machinery in reverse) versus a purpose-built protocol. The JSONL tool
  protocol already exists and may cover more of this than a fresh design would.
  Interacts with ADR-017's still-open question of where the protocol spec
  lives — recommendation unchanged: a tech spec, authored before the first
  authority item migrates.
- **OQ-3** — Trust model at the RPC boundary: which surfaces may assert which
  capabilities, and how "no gun and bullets together" is encoded once the
  caller is no longer the trusted process.
