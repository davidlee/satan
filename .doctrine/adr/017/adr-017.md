# ADR-017: Emacs is a client: lifecycle authority outside the editor, trust boundary as protocol

> **Imported 2026-07-22** from the `.emacs.d` doctrine corpus, frozen for SATAN.
> Ids cited below resolve in that corpus unless they also exist here; `DE-`/`DR-`
> have no doctrine equivalent, and `IMPR-NNN` / `ISSUE-NNN` are pre-doctrine
> prefixes for `IMP-NNN` / `ISS-NNN`.

> **Status: proposed.** Frames the end-state that RFC-001 (rehoming) and
> RFC-016 (evolution) both assume. Acceptance is the keeper's call; this ADR
> exists so that decision is recorded rather than made implicitly by smaller
> ones.

## Context

SATAN incubated inside the Emacs process: the broker owns lifecycle,
scheduling, context assembly, tool dispatch, validation, and audit.
`architecture.md` states "Emacs is the trusted local broker"; the trust
boundary is today a **process property** — safety holds because untrusted
output passes through broker code before touching the world.

Four forces have outgrown that framing:

1. **POL-001 has a test but no destination.** The earns-the-seat test decides
   module-by-module what leaves the editor; without a stated end-state,
   per-module calls converge to whatever topology the sequence of local
   decisions happens to produce (RFC-001 §D1).
2. **ADR-002 already moves the clock.** The metabolic arrival gate is placed
   in `satan-attrd` and is, by its own D2 resolution, "the system's only
   periodic clock." Whoever owns the clock owns the lifecycle. Accepting
   ADR-002 without this ADR decides the inversion silently, inside a
   scheduling decision, against the current architecture.md text.
3. **The pi-harness experience.** Running SATAN under the pi harness was
   useful with Emacs nowhere in the loop as host — direct evidence the
   editor's runtime role is incidental, not essential.
4. **The DNA thesis already implies it.** `governance.md`: SATAN's identity
   is durable DNA (ROM prompt, memory, permission model, constraints,
   protocol, invocation, audit); models, harnesses, *and the broker* are
   interchangeable. An interchangeable broker cannot be the definition of
   the trust boundary.

## Decision

**We will treat Emacs as SATAN's privileged client, not its host.**

1. **End-state topology.** SATAN's runtime is a headless core — daemons
   (satan-attrd and POL-001 successors), postgres, and the invocation
   policy. Emacs owns the editor-as-editor surfaces permanently: org
   substrate, daily block, denote/dired tools, interactive commands, and —
   because the keeper lives there — the human approval flows.

2. **The trust boundary becomes a protocol property.** Safety is defined by
   invariants any surface must satisfy — schema-validated actions, mode/tool
   allowlists, append-only audit, token ceilings, kill switches — not by
   which process the enforcement code runs in. The broker's current
   enforcement is one implementation of that protocol, not its definition.

3. **Authority inventory with single ownership.** Every item of authority
   (budget ceiling, allowlists, audit append, permission profiles, kill
   switches, arrival policy) has **exactly one owner at any time**, recorded
   in a ledger as it migrates. No item is enforced in two places; no item
   moves without its audit trail moving with it.

4. **Daemons are safe with the client absent.** No daemon may depend on the
   Emacs client for a safety property. If Emacs is not running, daemons
   idle, degrade, or enforce their ceilings alone — they never fall open.

5. **Migration is incremental, gated by POL-001 triggers.** This ADR names
   the destination; it schedules nothing. The earns-the-seat test continues
   to decide *what stays elisp*; this ADR decides *what the elisp is* — a
   client. The DNA thesis is what makes incremental rehosting cheap: the
   identity was never the elisp.

## Consequences

### Positive

- POL-001 extractions converge on a named topology instead of an accidental
  one; each extraction ADR/design cites a destination.
- ADR-002's daemon placement becomes a principled consequence rather than an
  anomaly against "Emacs is the trusted local broker."
- Crash-domain decoupling: editor freezes stop being agent-runtime outages
  and vice versa (the POL-001 motivating costs, resolved structurally).
- Harness-agnosticism strengthens: pi, CLI, or future surfaces are clients
  of the same protocol rather than emulations of the broker.
- RFC-001's composition root (the `satan` flake) gains a shape: core +
  clients + protocol, not "the emacs config plus satellites."

### Negative

- **Amends the DNA.** `architecture.md` "Emacs is the trusted local broker"
  and the Broker layer description require rewriting (amendment text below).
  Accepting this ADR requires that amendment, not a quiet override.
- **The protocol must be written down.** Today it exists implicitly in
  broker code (validators, allowlists, audit writers). A tech spec becomes
  mandatory before the first authority item migrates — new obligation.
- **Transition risk: split authority.** A half-migrated invariant enforced
  in two places (or neither) is worse than either end-state. Mitigated by
  the single-owner ledger (Decision §3).

### Neutral

- No immediate code change. Execution rides POL-001 triggers and RFC-001
  crystallization; SL-011 and RFC-016 phases proceed unchanged.
- The waybar/systemd wiring migrates naturally to the flake module (RFC-001
  D2) as arrival policy moves daemon-side (ADR-002).

## Amendment text (apply on acceptance)

`docs/satan/architecture.md`, replace the broker framing sentence:

> Emacs is SATAN's privileged client: it owns the canonical personal text
> substrate (org/denote) and every human approval surface. Authority over
> SATAN's actions is defined by protocol — validated actions, allowlists,
> append-only audit, ceilings, kill switches — enforced by whichever
> component owns each authority item (see the authority ledger). The model
> remains a reasoning engine, not trusted with direct authority; the trust
> boundary is the protocol, not a process.

`docs/satan/governance.md` core thesis, append one line after the DNA list:

> The broker is an implementation, not an identity: authority lives in the
> protocol and its ledger of owners (ADR-017).

## Verification

- **Authority ledger exists and is current** — every authority item names
  exactly one owner; reviewed at each extraction's audit.
- **Absent-client test** — daemons under test with no Emacs process enforce
  ceilings and fail closed.
- **No dual enforcement** — extraction audits check the migrated invariant
  is enforced at exactly one point (grep both sides).
- **Protocol spec landed before first authority migration** (tech spec, see
  Open questions).
