# RFC-001: SATAN exits the incubator: emacs as client, flake as composition root

> **Imported 2026-07-22** from the `.emacs.d` doctrine corpus, frozen for SATAN.
> Ids cited below resolve in that corpus unless they also exist here; `DE-`/`DR-`
> have no doctrine equivalent, and `IMPR-NNN` / `ISSUE-NNN` are pre-doctrine
> prefixes for `IMP-NNN` / `ISS-NNN`.

## Context

SATAN grew inside `~/.emacs.d` as an incubator: ~18.7k lines of elisp across 64
files in `satan/`, plus two satellites already spawned out (`~/dev/satan-attrd`,
Rust, ~7.3k lines; `~/dev/panopticon`, Python, ~3k lines), plus wiring scattered
through `~/flakes` (systemd units, home-manager), sway/waybar, postgres, and
state conventions under `~/.local/state/behaviour/`.

The incubation was deliberate and cheap — elisp put the substrate (org, buffers,
denote) at arm's reach and let the experiment prove itself before any
distributed-system investment. But the experiment has proven itself, and two
distinct architectural pressures are now conflated under one vague unease
("elisp is for emacs; 30k LOC of agent harness in an editor is a historical
accident"):

1. **Language/process boundary** — which modules belong in the editor process
   vs external daemons. Already governed: POL-001's earns-the-seat test
   ("does it use the editor as an editor?"), trigger-gated candidates
   (IMPR-006..009), active beachhead (`satan-attrd`), and ADR-001's
   perception/cognition split. This axis needs execution, not new decisions.

2. **Ownership boundary** — whether SATAN is *configuration* or a *project*.
   Ungoverned and unasked. Living inside `.emacs.d` means the editor config's
   build system is the agent runtime's build system: emacs-overlay parses it
   for `use-package` forms, compile-angel byte-compiles the runtime on save,
   flake visibility depends on git-tracked status, `just check` gates both.

A third pressure follows from the sprawl: **SATAN has no composition root**. No
single artifact answers "what is SATAN and how does it deploy" — the system is
defined by scatter across five subsystems and three repos.

Running SATAN under the pi harness settled the directional question
experientially: the system was useful with emacs nowhere in the loop as host.
Per the governance core thesis (docs/satan/governance.md), SATAN's identity is
its durable DNA — ROM prompt, self-authored memory, permission model, runtime
constraints, tool protocol, invocation schedule, audit trail — with models,
harnesses, and broker all interchangeable. That thesis quietly implies the
end-state this RFC makes explicit.

## Discussion

### D1 — End-state: emacs is a client, not the host

Two candidate destinations for the per-module extractions POL-001 gates:

- **Emacs as runtime** (status quo): daemons orbit the broker; broker owns
  lifecycle, scheduling, trust.
- **Emacs as client** (proposed): a headless core (Rust daemons + postgres +
  schedule) owns lifecycle; emacs provides surfaces, org integration, and
  human approval. The trust boundary becomes a protocol property, not a
  process property.

Position: **client**. Evidence: the pi-harness experience; the DNA thesis
(identity survives rehosting by design); ADR-001 already moving cognition out
of the tick; every POL-001 extraction candidate trending the same direction.
Without a stated destination, per-module trigger decisions converge to whatever
topology the sequence of local calls happens to produce.

This does not diminish the editor's role — the modules that use the editor *as
an editor* (org-block writer, denote/dired tools, context assembly over org
files, interactive surfaces, approval flows) keep their seats permanently.
They become the client.

### D2 — Nix flake as composition root

A `satan` flake, top-level coordinator for the whole organism, exposing:

- the emacs package (broker, surfaces, org tools — the client);
- the Rust daemon crates (satan-attrd and successors);
- schema/migration ownership (see D4);
- a home-manager module wiring systemd units, the waybar widget, and postgres
  provisioning.

`~/flakes` then consumes **one input** instead of hand-assembling five
subsystems. The system stays multi-process and multi-language but gains a
single definition. Sprawl becomes declared dependencies.

### D3 — SATAN's elisp becomes an external package

The ownership cut, orthogonal to and much cheaper than any language cut:
extract `satan/` from `.emacs.d` into its own repo (or the flake repo of D2),
consumed by the config as a package. Precedent: magit is 25k+ lines of elisp
and is unambiguously legitimate — because it is a package, not someone's
config. The child keeps being elisp where elisp earned its seat; it stops
being *inside* the config. `.emacs.d` shrinks back to configuration.

This also defuses the config-specific traps (flake sees only git-tracked
files, emacs-overlay parse coupling, compile-angel blast radius) for the
runtime mass.

### D4 — Schema ownership consolidates

Current state is an early fragmentation warning: two migration runners with
colliding numbering. The elisp runner (`dl-satan-memory-migrate.el`) owns
`satan_memory` (0001–0007: memory, patch_jobs, interventions, patterns);
satan-attrd owns `satan_attributes` (0007–0012). Separate databases, so not
broken — but "0007" already means two different things, and each future daemon
extraction would bring its own migration dialect.

Position: one schema owner per database, migrations owned at the flake level
or by the DB-owning component — not by whichever process happened to create a
table first. When IMPR-007 (memory substrate → `satan-memoryd`) fires, the
elisp migration runner and most of the ten-file psql surface leave with it.
(Shelling out to psql from elisp works — that is the most that can be said
for it — and it needs no remediation ahead of IMPR-007.)

### D5 — Interface style: durable logs over RPC

Panopticon is the in-house proof of the right boundary pattern: producers →
durable JSONL → derived data → consumers that merely read. Files as the
interface; zero coupling to emacs internals; aged well. Extractions should
prefer this shape (segment logs, cursors — cf. ADR-001's parcel-as-view) over
chatty RPC between broker and daemons wherever latency allows.

### Alternatives considered

- **Status quo** (POL-001 per-module gating only): sound pacing, but converges
  to an accidental topology; leaves the ownership axis and composition root
  unaddressed indefinitely.
- **Full rewrite** (headless core now, big-bang): discards a working system;
  the DNA thesis makes it unnecessary — identity rehosts incrementally.
- **Repo extraction without a flake root**: fixes ownership but leaves the
  system defined by scatter; half the value.

### Crystallization path

As positions harden, extract:

- **ADR**: end-state — emacs as client; trust boundary as protocol property
  (D1). Amends the framing of POL-001's "broker IS the trust boundary" without
  contradicting its module test.
- **ADR or policy**: composition root + schema ownership rules (D2, D4).
- **Backlog items**: repo extraction of `satan/` (D3); author the `satan`
  flake + home-manager module (D2); consolidate migration numbering /
  ownership (D4).
- POL-001 itself survives unchanged as the module-level test; its candidate
  list continues to gate the language-axis work.

## Outcome

Open. Direction (client end-state, flake root, external package) is agreed in
principle; unresolved:

- sequencing of the ownership cut (D3) relative to in-flight language cuts
  (IMPR-003 beachhead, IMPR-006 patch runner) — extract repo first, or land
  the beachhead first?
- whether the emacs package lives in the `satan` flake repo or its own;
- migration renumbering vs freeze-and-diverge for the two existing sequences.
